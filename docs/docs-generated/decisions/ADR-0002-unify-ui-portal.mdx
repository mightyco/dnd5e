# ADR-0002: Unify UI Portal and Decouple API Layer

## Context and Problem Statement

Currently, the Simulation Dashboard is embedded deep within the documentation site (Docusaurus) and relies on a separate local Sinatra API (`sim_server.rb`) to run simulations. This architecture creates a "mish-mash" where the UI feels like a documentation page rather than a standalone application. Furthermore, running two separate servers (Node for the UI, Ruby for the API) locally creates friction and connection issues. How can we redesign the architecture to make the Simulation Portal a distinct, unified application while keeping the documentation as a supportive sub-section?

## Decision Drivers

*   The user experience: The portal should feel like a dedicated application, not a sub-page of the docs.
*   Operational simplicity: Reduce the need to run and manage multiple concurrent local servers.
*   Architectural clarity: The separation of the `sim_server.rb` API is currently unnecessary overhead for local use.
*   Maintainability of "Math Transparency" and simulation analysis tools.

## Considered Options

*   **Option 1: Compile Ruby to WebAssembly (Wasm)**. Run the D&D engine entirely in the browser using ruby.wasm, eliminating the API server completely.
*   **Option 2: Unified Ruby Web App (Rails or Sinatra/React)**. Move the React frontend into a unified Ruby web application (e.g., using ViteRuby or Sinatra serving static React build).
*   **Option 3: Electron or Tauri Desktop App**. Bundle the Ruby engine and the React UI into a single cross-platform desktop application.
*   **Option 4: Keep Docusaurus but invert the hierarchy**. Make the React application the primary entry point and serve the Docusaurus build statically via an iframe or sub-route.

## Pros and Cons of the Options

### Option 1: Compile Ruby to WebAssembly (Wasm)

* <span className="consequence-keyword good">Good</span>, because it completely eliminates the need for an API server; the entire app runs client-side.
* <span className="consequence-keyword good">Good</span>, because deployment becomes trivial (pure static hosting).
* <span className="consequence-keyword bad">Bad</span>, because compiling complex Ruby dependencies (like pdf-reader) to Wasm might require significant refactoring.

### Option 2: Unified Ruby Web App (Rails or Sinatra/React)

* <span className="consequence-keyword good">Good</span>, because it aligns with the core Ruby focus of the project.
* <span className="consequence-keyword good">Good</span>, because it's a well-understood architecture (Single server serving API and static assets).
* <span className="consequence-keyword bad">Bad</span>, because it abandons the "Design-as-Code" Docusaurus portal we just built.

### Option 3: Electron or Tauri Desktop App

* <span className="consequence-keyword good">Good</span>, because it provides a true "standalone app" experience without browser CORS/port issues.
* <span className="consequence-keyword bad">Bad</span>, because it introduces significant packaging and distribution overhead.

## Math Transparency (D&D 2024 Project)

Regardless of the chosen option, the JSON payload generated by the simulator will remain the source of truth for the Roll Inspector. The architecture must ensure that the full raw roll data and modifier calculations are preserved when crossing the boundary between the Ruby engine and the JS/TS frontend.
